# 在 Claude Code 中設定

請按照以下步驟設定開發流程系統，設定完成後，你的專案結構應該是這樣：

```Plain
project/
 Product-Spec.md                      # 產品需求文件 (自動生成)
 Product-Spec-CHANGELOG.md            # 變更記錄 (自動生成)
 UI-Prompts.md                        # 原型圖提示詞 (自動生成)
 .claude/
     CLAUDE.md                        # 主控設定 (必須建立)
     skills/                          # 技能包 (必須建立)
         product-spec-builder/        # 產品經理技能包
            SKILL.md
            templates/
                product-spec-template.md
                changelog-template.md
         ui-prompt-generator/         # UI 提示詞設計師技能包
            SKILL.md
            templates/
                ui-prompt-template.md
         dev-builder/                 # 全端開發工程師技能包
             SKILL.md
```

## 建立主控設定
1. 在 .claude/ 目錄下建立 CLAUDE.md 檔案
2. 貼上主控角色定義、工作流程、指令集
3. 包含專案狀態偵測、技能呼叫規則、四個階段的流程定義（需求收集 -> 原型設計 -> 專案開發 -> 本地執行）

## 建立產品經理技能包
4. 在 .claude/skills/ 目錄下建立 product-spec-builder/ 資料夾
5. 建立 SKILL.md 定義毒舌產品經理角色、追問策略、0-1 模式和迭代模式的切換邏輯
6. 在 templates/ 下建立 product-spec-template.md 定義產品文件輸出格式
7. 在 templates/ 下建立 changelog-template.md 定義變更記錄格式

## 建立 UI 提示詞設計師技能包
8. 在 .claude/skills/ 目錄下建立 ui-prompt-generator/ 資料夾
9. 建立 SKILL.md 定義提示詞生成規則、視覺風格選項、配色方案
10. 在 templates/ 下建立 ui-prompt-template.md 定義提示詞輸出格式

## 建立全端開發工程師技能包
11. 在 .claude/skills/ 目錄下建立 dev-builder/ 資料夾
12. 建立 SKILL.md 定義技術棧選擇策略、專案初始化流程、程式碼實作規範

## 重要設定細節
- 技能呼叫規則：CLAUDE.md 定義了調度規則，需求收集呼叫產品經理，原型設計呼叫 UI 設計師，程式碼開發呼叫全端工程師
- 模式自動切換：啟動時自動偵測 Product-Spec.md 是否存在，沒有則進入 0-1 模式，有則進入迭代模式
- 循環配合機制：每次新增或修改功能，先呼叫產品經理更新文件，再呼叫開發實作程式碼，確保文件和程式碼同步
- 範本約束：templates/ 中的格式是強制性的，所有輸出必須按照範本結構
- 文件驅動：所有產物寫入專案根目錄，對話中斷後可透過 /status 恢復進度
- 指令集：支援 /prd (需求收集)、/ui (生成原型圖提示詞)、/dev (開發程式碼)、/run (本地執行)、/check (檢查完整度)

# 核心設計理念

```Plain

  CLAUDE.md (主控)                                                         <- 流程層：控制開發流程
                                                                             定義工作流、指令集、技能調度規則

  Skills (技能包)                                                          <- 執行層：分工協助
  |- product-spec-builder       收集需求、生成產品文件、迭代更新              
  |- ui-prompt-generator        生成原型圖提示詞                           
  |- dev-builder                選擇技術棧、搭建專案、實作程式碼              

  Templates (範本)                                                         <- 知識層：提供輸出規範
  |- product-spec-template      產品文件格式、章節結構、範例                  
  |- changelog-template         變更記錄格式、版本號規則                     
  |- ui-prompt-template         提示詞結構、風格選項、品質詞                  

  輸出檔案                                                                 <- 文件層：結構化儲存
  |- Product-Spec.md            產品需求文件                               
  |- Product-Spec-CHANGELOG.md  變更記錄                                  
  |- UI-Prompts.md              原型圖提示詞                               

```

## 工作流程
1. 說出想法：告訴 AI 你想做什麼產品
2. 需求收集：/prd 產品經理透過追問完善需求細節
3. 生成文件：自動生成 Product-Spec.md 和 CHANGELOG
4. 原型設計：/ui 根據產品文件生成原型圖提示詞
5. 生成原型圖：把提示詞發給 Gemini 3.1 生成設計稿
6. 前端開發：用 Google AI Studio Builder 開發前端
7. 後端改造：/dev 把前端程式碼改造成前後端分離專案
8. 本地執行：/run 啟動專案，驗證功能
9. 功能迭代：提出修改需求，產品經理先更新文件，開發再實作程式碼
10. 完整度檢查：/check 對照產品文件檢查功能是否遺漏

## 關鍵特性
- Agent + Skill 解耦：主控負責調度，技能負責專業能力，可獨立替換
- 0-1 和迭代模式自動切換：啟動時偵測產品文件是否存在，自動進入對應模式
- 毒舌追問機制：不接受模糊回答，發現邏輯漏洞直接指出，逼使用者想清楚
- 循環配合流程：每次修改先更新文件再寫程式碼，文件和程式碼始終同步
- 範本約束：嚴格遵循 templates/ 定義的格式，保證輸出規範統一
- AI 增強建議：產品經理會主動建議用 AI 簡化繁瑣的手動流程
- 衝突偵測：迭代模式下自動偵測新需求和現有文件的衝突
- 文件驅動：所有產物寫入檔案，對話中斷後可透過 /status 恢復進度
- 專案狀態偵測：啟動時自動掃描目錄，偵測進度並路由到對應階段

# 模型選擇說明
這套方案依賴 Claude Code (Sonnet 4.6 / Opus 4.6) 的 Agent + Skill 功能來實作自動化開發流程。如果你用不了 Claude Code，也可以在 OpenCode 中使用，模型可以選擇 GPT-5.3。
需要注意的是，不同模型在理解需求、執行複雜指令、生成高品質程式碼等方面的能力存在差異。從實測效果來看：
- 理解能力：模型對使用者需求的理解深度會影響產品文件的品質，例如能否準確識別核心功能、判斷功能優先順序、發現需求漏洞
- 執行能力：模型遵循範本格式、按照技能定義執行任務的嚴格程度會影響輸出品質
- 程式碼品質：模型在生成程式碼時的規範性、可讀性、錯誤處理能力會影響最終專案的可用性
- 多輪協助：模型在產品經理和開發工程師循環配合過程中保持內容一致的能力會影響迭代效率
建議選擇支援長內容的模型。初次使用時可以先用一個簡單專案測試效果，確認該模型能否滿足你的品質要求，再進行複雜專案。如果發現生成內容偏離預期較多，可以嘗試更換模型或調整設定。
